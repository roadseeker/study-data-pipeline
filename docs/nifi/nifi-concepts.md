# NiFi 핵심 개념 정리

## 문서 목적

이 문서는 Nexus Pay Week 3 실습에서 사용할 Apache NiFi의 핵심 개념을 정리한 컨설팅 설명 자료다.
목표는 REST API, CSV 파일, PostgreSQL 데이터를 하나의 수집 파이프라인으로 통합하기 전에
NiFi의 처리 단위와 운영 개념을 명확히 이해하는 것이다.

## 적용 시나리오

Nexus Pay는 다음 세 가지 이기종 소스를 수집해야 한다.

- 결제 API: 실시간 JSON 이벤트
- 정산 CSV: 레거시 시스템이 주기적으로 생성하는 파일
- 고객 마스터 DB: PostgreSQL의 변경 데이터

NiFi는 이 세 소스를 수집하고, 표준 스키마로 정리한 뒤 Kafka로 전달하며,
각 데이터가 언제 어디서 들어왔는지 Provenance로 추적하는 역할을 맡는다.

고객 요구사항을 NiFi 관점으로 해석하면 다음과 같다.

- 여러 소스를 하나의 수집 파이프라인으로 통합해야 한다.
- 소스별 형식 차이를 표준 스키마로 정리해야 한다.
- 장애나 품질 문제 시 어느 단계에서 실패했는지 확인 가능해야 한다.
- 감사 대응을 위해 데이터 출처와 변환 경로를 추적할 수 있어야 한다.

즉 Week 3에서 NiFi의 역할은 단순 수집이 아니라
"이기종 소스 통합 + 데이터 표준화 + 운영 가시성 + 감사 추적"을 제공하는 것이다.

## 1. FlowFile

FlowFile은 NiFi에서 처리되는 데이터의 최소 단위다.
하나의 FlowFile은 다음 두 부분으로 구성된다.

- Content: 실제 데이터 본문
- Attributes: 파일명, MIME 타입, 소스 식별자, 처리 시각 같은 메타데이터

Nexus Pay 기준 예시는 다음과 같다.

- API 호출 결과 JSON 한 건 또는 한 묶음
- 정산 CSV 파일 한 개
- PostgreSQL 조회 결과 레코드 집합

실무 포인트:

- Content와 Attributes를 분리해 관리하므로, 본문을 바꾸지 않고도 라우팅과 추적이 가능하다.
- 데이터 계보 추적 시 어떤 소스에서 어떤 속성으로 들어왔는지 확인하기 쉽다.

## 2. Processor

Processor는 FlowFile을 생성, 변환, 검증, 전달하는 작업 단위다.
NiFi 플로우는 여러 Processor를 연결해 구성한다.

Nexus Pay Week 3에서 사용할 대표 Processor는 다음과 같다.

- `InvokeHTTP`: 결제 API 호출
- `ListFile` / `FetchFile`: 정산 CSV 탐지 및 읽기
- `QueryDatabaseTable`: 고객 마스터 증분 조회
- `ConvertRecord`: CSV 또는 Avro를 JSON으로 변환
- `JoltTransformJSON`: 표준 스키마로 구조 변환
- `ValidateRecord`: 필수 필드와 형식 검증
- `PublishKafka_2_6`: Kafka 토픽 전송

실무 포인트:

- 각 Processor는 단일 책임으로 설계하는 것이 운영과 장애 분석에 유리하다.
- 소스 수집, 변환, 검증, 전달을 분리하면 병목 지점과 실패 구간이 명확해진다.

컨설팅 설명 포인트:

- API, 파일, DB를 하나의 거대한 로직으로 처리하지 않고 단계별 Processor로 나누면
  고객은 데이터 흐름을 시각적으로 이해할 수 있다.
- 운영자는 어느 단계에서 실패했는지 빠르게 확인할 수 있고,
  변경 범위도 특정 Processor 단위로 제한할 수 있다.

## 3. Connection과 Back-Pressure

Connection은 Processor와 Processor 사이를 연결하는 큐다.
FlowFile은 Connection에 잠시 저장되었다가 다음 Processor로 전달된다.

Connection의 핵심 기능은 다음과 같다.

- Relationship 기반 라우팅
- 큐 길이 시각화
- Back-Pressure 제어

Back-Pressure는 큐에 FlowFile 수나 데이터 크기가 일정 수준 이상 쌓이면
상류 Processor를 자동으로 일시 정지시켜 전체 시스템을 보호하는 기능이다.

Nexus Pay 관점에서 중요한 이유는 다음과 같다.

- Kafka 장애 또는 지연 발생 시 NiFi가 무한정 밀어 넣지 않도록 제어 가능
- DB/파일/API 소스 속도와 Kafka 전달 속도가 달라도 완충 지점을 둘 수 있음
- 운영자가 어느 구간에서 적체가 발생하는지 빠르게 파악 가능

컨설팅 설명 포인트:

- 금융 시스템은 "데이터를 많이 받는 것"보다 "안정적으로 받는 것"이 더 중요하다.
- Back-Pressure는 처리량 초과 상황에서 전체 파이프라인이 무너지지 않도록 막는 안전장치다.

## 4. Process Group

Process Group은 여러 Processor를 논리적으로 묶는 컨테이너다.
복잡한 플로우를 기능별, 소스별로 나눠 관리할 수 있다.

Week 3에서 설계할 Nexus Pay 그룹 구조는 다음과 같다.

- `PG-1: API Ingestion`
- `PG-2: File Ingestion`
- `PG-3: DB Ingestion`
- `PG-4: Schema Standardization`
- `PG-5: Kafka Publishing`

그룹 간 관계는 다음과 같다.

```text
PG-1 API Ingestion --------\
PG-2 File Ingestion --------> PG-4 Schema Standardization --> PG-5 Kafka Publishing
PG-3 DB Ingestion ---------/
```

즉 `PG-1`, `PG-2`, `PG-3`은 서로 다른 원천 데이터를 가져오는
독립적인 입력 계층이고,
`PG-4`는 이 세 소스가 공통으로 거치는 표준화·품질 검증 계층이며,
`PG-5`는 최종 Kafka 전달 계층이다.

조금 더 구체적으로 보면 다음과 같다.

- `PG-1: API Ingestion`
  결제 API에서 JSON 데이터를 수집한 뒤 후속 표준화 단계로 넘긴다.
- `PG-2: File Ingestion`
  정산 CSV 파일을 읽고 처리 가능한 구조로 바꾼 뒤 표준화 단계로 넘긴다.
- `PG-3: DB Ingestion`
  PostgreSQL 고객 마스터 데이터를 증분 조회한 뒤 표준화 단계로 넘긴다.
- `PG-4: Schema Standardization`
  세 소스에서 들어온 데이터를 하나의 공통 이벤트 구조로 맞추고 품질을 검증한다.
- `PG-5: Kafka Publishing`
  표준화와 검증을 통과한 데이터를 Kafka 토픽으로 전송한다.

이 구조의 핵심은 "소스별 수집"과 "공통 후처리"를 분리하는 데 있다.

- 수집 로직은 소스별로 독립적으로 관리할 수 있다.
- 표준화와 검증 규칙은 한 곳에서 공통 적용할 수 있다.
- 한 소스에 장애가 생겨도 다른 소스의 수집 흐름과 분리해 대응하기 쉽다.
- Provenance로도 어느 소스에서 들어와 어떤 공통 단계를 거쳐 Kafka로 갔는지 추적하기 쉽다.

실무 포인트:

- 그룹 간 Input Port / Output Port를 사용하면 플로우 경계가 명확해진다.
- 장애 분석 시 어느 단계에서 실패했는지 빠르게 좁힐 수 있다.
- 추후 소스가 늘어나도 동일 패턴으로 확장하기 쉽다.

컨설팅 설명 포인트:

- 고객에게 NiFi 화면을 보여줄 때 Process Group 구조가 곧 아키텍처 설명 자료가 된다.
- 소스별 수집과 공통 표준화/전달 단계를 분리하면 운영 책임과 변경 영향을 명확히 나눌 수 있다.

## 5. Schema Standardization

Schema Standardization은 서로 다른 원천 데이터 구조를 공통 형식으로 맞추는 단계다.
Week 3에서는 API, CSV, DB 데이터가 모두 다른 필드 구조를 가지므로
Kafka 적재 전 하나의 표준 이벤트 구조로 정규화해야 한다.

Nexus Pay 기준 핵심 목적은 다음과 같다.

- 소스가 달라도 downstream 시스템이 같은 필드 구조로 읽을 수 있게 함
- 필수 필드 누락, 타입 불일치 같은 품질 문제를 조기에 발견
- Kafka, Flink, Spark, Airflow가 동일한 이벤트 계약을 사용하게 함

대표적으로 사용할 Processor는 다음과 같다.

- `JoltTransformJSON`: JSON 구조 변환
- `UpdateAttribute`: 소스 태그, 수집 시각, 추적용 메타데이터 추가
- `ValidateRecord`: 표준 스키마 적합성 검증

실무 포인트:

- 표준화 단계가 없으면 소스가 늘어날수록 후속 시스템이 복잡해진다.
- 표준 스키마는 수집 파이프라인의 기술 표준이자 운영 계약이다.

## 6. Provenance

Provenance는 FlowFile의 생성, 수정, 라우팅, 전송 이력을 기록하는 추적 기능이다.
NiFi의 감사 대응과 운영 가시성에서 가장 중요한 기능 중 하나다.

Nexus Pay에서 Provenance가 중요한 이유는 다음과 같다.

- 특정 결제 이벤트가 API, CSV, DB 중 어디서 들어왔는지 증명 가능
- 어떤 Processor를 거쳐 Kafka로 전송되었는지 확인 가능
- 장애나 데이터 품질 문제 발생 시 정확한 변환 경로를 역추적 가능
- 금융감독원 감사 대응용 계보(Lineage) 근거로 활용 가능

컨설팅 설명 포인트:

- Provenance는 "문제가 생기면 찾아보는 로그"가 아니라
  "데이터 출처를 증명하는 감사 기록"으로 설명하는 것이 맞다.
- 고객이 요구한 "어떤 데이터가 언제 어디서 들어왔는지 추적 가능해야 한다"는 요구에
  직접 대응하는 NiFi의 핵심 기능이다.

예시 질문:

- `PAY-00000042` 이벤트는 어느 시각에 API에서 수집되었는가?
- 이 이벤트는 어떤 Jolt 변환을 거쳐 Kafka로 전달되었는가?
- 왜 특정 레코드는 정상 토픽이 아니라 DLQ로 이동했는가?

## 7. NiFi를 사용하는 이유

Nexus Pay Week 3에서 NiFi를 사용하는 이유는
단순히 데이터를 한 번 가져오는 스크립트를 만드는 것이 아니라,
실제 운영 환경에서 계속 돌릴 수 있는 수집 파이프라인을 만들기 위해서다.

여기서 "운영 가능한 데이터 흐름 플랫폼"이라는 말은 다음 뜻이다.

- 데이터를 수집할 수 있다.
- 데이터를 표준 형식으로 변환할 수 있다.
- 장애가 나면 어디서 막혔는지 볼 수 있다.
- 실패 데이터를 재시도하거나 다른 경로로 보낼 수 있다.
- 데이터가 언제, 어디서, 어떤 단계를 거쳤는지 추적할 수 있다.

즉 NiFi는 단순한 "수집 스크립트 실행기"가 아니라
"수집, 변환, 검증, 라우팅, 추적, 운영"을 하나의 흐름으로 관리하는 플랫폼이다.

### 7-1. 단순 스크립트와 NiFi의 차이

예를 들어 Python 스크립트로도 다음 작업은 가능하다.

- API를 호출해서 JSON 받기
- CSV 파일 읽기
- PostgreSQL 조회하기
- 결과를 Kafka에 넣기

이 정도는 "기능 구현" 관점에서는 충분할 수 있다.
하지만 금융 데이터 파이프라인은 여기서 끝나지 않는다.

운영 환경에서는 다음 질문에 답할 수 있어야 한다.

- API가 잠시 느려지거나 실패하면 어떻게 처리할 것인가?
- 같은 CSV 파일을 중복 처리하지 않으려면 어떻게 할 것인가?
- DB에서 마지막으로 어디까지 읽었는지 어떻게 기억할 것인가?
- 잘못된 데이터는 정상 토픽이 아니라 어디로 보낼 것인가?
- 특정 거래가 언제 어떤 경로를 거쳐 Kafka까지 갔는지 어떻게 증명할 것인가?

NiFi는 이런 운영 질문에 답할 수 있는 기능을 기본 제공한다.

### 7-2. Nexus Pay 예시로 보는 이유

#### 예시 1. 결제 API 수집

단순 스크립트라면 `requests.get()`으로 API를 호출하고
성공하면 Kafka로 보내는 형태가 된다.

하지만 실제 운영에서는 이런 상황이 발생한다.

- API 응답이 일시적으로 느려짐
- 일부 응답 JSON이 깨져 있음
- 네트워크 타임아웃이 발생함

NiFi를 사용하면 다음처럼 대응할 수 있다.

- `InvokeHTTP`에서 성공/실패 경로를 분리
- 실패 건은 재시도 큐 또는 오류 경로로 분기
- 어느 시점부터 실패가 늘었는지 Bulletin과 큐 상태로 확인

즉 "API에서 데이터를 받는다"에서 끝나지 않고
"문제가 생겨도 운영자가 통제할 수 있는 형태"가 된다.

#### 예시 2. 정산 CSV 파일 수집

레거시 정산 시스템이 매시간 CSV 파일을 떨군다고 가정하면,
단순 스크립트는 주기적으로 폴더를 확인하고 파일을 읽을 수 있다.

하지만 운영에서는 이런 문제가 생긴다.

- 이미 읽은 파일을 또 읽으면 중복 정산 위험이 생김
- 파일 포맷이 깨졌을 때 전체 파이프라인이 흔들릴 수 있음
- 파일이 갑자기 많이 쌓이면 처리 속도 제어가 필요함

NiFi를 사용하면 다음이 가능하다.

- `ListFile` / `FetchFile`로 읽은 파일과 새 파일을 관리
- 오류 파일은 실패 경로로 분리
- Connection 큐와 Back-Pressure로 적체 구간을 시각적으로 확인

즉 파일 수집이 "폴더 읽기"가 아니라
"중복 방지와 적체 관리가 가능한 운영 파이프라인"이 된다.

#### 예시 3. PostgreSQL 고객 마스터 수집

고객 마스터 데이터를 단순 스크립트로 읽으면
매번 전체 테이블을 다시 읽는 방식이 되기 쉽다.

이 경우 운영상 문제가 생긴다.

- DB 부하가 커짐
- 마지막 읽은 위치를 애플리케이션이 직접 관리해야 함
- 장애 후 재시작 시 어디서부터 이어야 하는지 복잡해짐

NiFi의 `QueryDatabaseTable`을 사용하면
증분 컬럼 기준으로 마지막 읽은 지점을 관리하면서
계속 이어서 수집할 수 있다.

즉 DB 수집이 "매번 전체 조회"가 아니라
"운영 중 이어서 읽는 증분 수집"으로 바뀐다.

### 7-3. 감사 대응에서 왜 중요한가

고객 요구사항에서 가장 중요한 부분은 추적 가능성이다.
금융감독원 감사 상황에서는 단순히 "데이터를 받았다"가 아니라
"이 데이터가 어디서 왔고 어떤 변환을 거쳐 저장되었는가"를 설명할 수 있어야 한다.

NiFi의 `Provenance`는 바로 이 부분을 지원한다.

예를 들어 `PAY-00012345`라는 거래 이벤트에 대해
다음 같은 질문에 답할 수 있다.

- 이 데이터는 API, CSV, DB 중 어디서 들어왔는가?
- 몇 시에 수집되었는가?
- 어떤 Processor를 거쳐 표준 스키마로 변환되었는가?
- 왜 정상 Kafka 토픽이 아니라 DLQ로 갔는가?

즉 Provenance는 단순 로그가 아니라
데이터 출처와 변환 경로를 증명하는 감사 기록이다.

### 7-4. 한 문장으로 정리

NiFi를 사용하는 이유는 데이터를 "가져오기 위해서"가 아니라
여러 소스의 데이터를 안정적으로 수집하고, 표준화하고, 문제 발생 시 통제하고,
감사 대응이 가능하도록 전체 흐름을 운영하기 위해서다.

## 8. 고객 요구사항과 NiFi 기능 매핑

| 고객 요구사항 | NiFi 기능 | Week 3 적용 방식 |
|------|------|------------------|
| API, CSV, DB를 하나의 흐름으로 통합 | Process Group, Input/Output Port | 소스별 수집 그룹 + 공통 표준화/전달 그룹 구성 |
| 형식이 다른 데이터를 공통 구조로 정리 | JoltTransformJSON, ValidateRecord | 표준 이벤트 스키마로 통합 |
| 장애 시 어느 단계에서 막히는지 확인 | Connection, Back-Pressure, Bulletin | 큐 적체와 실패 지점 가시화 |
| 감사 시 데이터 출처 증명 | Provenance | 수집 시각, 변환 경로, Kafka 전달 이력 추적 |
| Kafka로 실시간 전달 | PublishKafka_2_6 | 정상 데이터와 DLQ를 분리 전송 |

## 9. Week 3 실습 연결 포인트

이 문서를 바탕으로 Week 3에서는 다음 순서로 진행한다.

1. API, 파일, DB 수집용 Process Group 설계
2. 각 소스별 Processor 체인 구성
3. JoltTransformJSON으로 표준 스키마 통일
4. ValidateRecord로 품질 검증
5. PublishKafka로 Kafka 전달
6. Provenance로 전체 흐름 추적

## 10. 요약

- FlowFile: NiFi가 다루는 데이터 단위
- Processor: 데이터를 생성, 변환, 전달하는 작업 단위
- Connection: Processor 사이의 큐와 라우팅 경로
- Back-Pressure: 과부하 방지 장치
- Process Group: 복잡한 플로우를 구조화하는 논리 그룹
- Provenance: 데이터 계보와 감사 추적의 핵심 기능

Nexus Pay Week 3의 목표는 이 개념들을 바탕으로
다중 소스 데이터를 표준화하여 Kafka로 전달하고, 전체 경로를 추적 가능한 형태로 구현하는 것이다.
