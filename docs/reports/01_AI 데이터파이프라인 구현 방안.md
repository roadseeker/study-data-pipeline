# 01_AI 데이터파이프라인 구현 방안

## 1. 문서 목적

이 문서는 특정 도메인 전용이 아닌 `범용 오픈소스 AI 데이터파이프라인 플랫폼`을 구축하기 위한 방향을 정리한 제안서다.

목표는 다음과 같다.

- 자연어 목표를 데이터 파이프라인 명세로 변환
- 다양한 소스와 타겟을 연결하는 파이프라인을 자동 또는 반자동으로 생성
- Kafka, NiFi, Flink, Spark, Airflow, DB, Redis를 조합한 실행 구조 제공
- MCP 서버를 통해 각 시스템을 AI가 읽고 다룰 수 있게 표준화
- OpenClaw, LibreChat, Open WebUI 같은 오픈소스 AI 인터페이스와 연결
- 사람 승인과 검증을 포함한 안전한 운영 자동화 체계 구축

이 문서는 현재 저장소의 Nexus Pay 실습 자산을 참조 시나리오로 활용하지만, 플랫폼 설계 자체는 고객·도메인 독립적으로 재사용 가능한 방향을 전제로 한다.

## 2. 핵심 결론

범용 AI 데이터파이프라인은 충분히 구현 가능하다. 다만 성공하려면 "AI가 즉흥적으로 모든 것을 생성하는 방식"이 아니라 아래 구조로 접근해야 한다.

- 자연어 목표 -> 구조화된 파이프라인 spec
- spec -> template / adapter 기반 인프라 및 파이프라인 생성
- 생성 결과 -> 검증 및 테스트
- 운영 단계 -> MCP 서버를 통한 상태 조회와 설명
- 승인 이후 -> 실행, 변경, 재실행

핵심은 모델 자체보다 다음 자산이다.

- pipeline spec
- template engine
- source/sink adapter
- MCP server
- validation layer
- approval workflow

## 3. 범용 플랫폼 관점의 상위 아키텍처

### 3-1. 상위 개념

```text
사용자 목표
  -> AI Planner / Spec Builder
  -> Pipeline Spec
  -> Template / Adapter Engine
  -> Kafka / NiFi / Flink / Spark / Airflow / DB / Redis
  -> Validator
  -> MCP Servers
  -> OpenClaw / LibreChat / Open WebUI / Admin Portal
```

### 3-2. 계층별 역할

| 계층 | 역할 |
|------|------|
| Interaction Layer | OpenClaw, LibreChat, Open WebUI, 향후 Admin Portal |
| Planning Layer | 자연어 분석, spec 생성, 작업 분해 |
| Control Layer | MCP 서버, policy, approval, tool routing |
| Execution Layer | Kafka, NiFi, Flink, Spark, Airflow, DB, Redis |
| Validation Layer | 품질 검증, 정합성 확인, 상태 판정, DLQ, 테스트 |

## 4. OpenClaw의 위치

OpenClaw는 이 플랫폼에서 "모델" 자체가 아니라 `상위 실행 콘솔` 또는 `에이전트 런타임` 역할로 두는 것이 적절하다.

즉 OpenClaw는 다음을 담당한다.

- 자연어 요청 수신
- pipeline spec 초안 생성
- MCP 도구 호출
- shell/file/git 실행 워커 호출
- 검증 결과 요약
- 승인 전 변경 diff 제시

이 구조가 좋은 이유는 OpenClaw가 다음과 같은 요구를 처리하기에 적합하기 때문이다.

- "이 CSV 폴더를 표준 이벤트 스키마로 Kafka에 적재해"
- "NiFi 수집 플로우 초안 만들고 검증해"
- "Kafka ISR 감소 원인을 설명해"
- "Airflow DAG를 생성하고 실행 전 테스트해"
- "Flink job 상태를 확인하고 checkpoint 실패 원인을 요약해"

즉 OpenClaw는 범용 AI 데이터파이프라인의 상위 제어 진입점으로 보는 것이 맞다.

## 5. 왜 오픈소스 중심이어야 하는가

상용 코딩 에이전트에 의존하지 않고 오픈소스 중심으로 가는 이유는 다음과 같다.

- 특정 벤더 종속성 감소
- 온프레미스 또는 폐쇄망 배포 가능
- 고객 환경 맞춤형 확장 가능
- 장기적으로 플랫폼 자산을 직접 축적 가능
- MCP, spec, adapter 자산을 재사용 가능

### 5-1. 추천 오픈소스 구성

| 영역 | 후보 |
|------|------|
| 상위 실행 콘솔 | OpenClaw |
| 대화형 UI | LibreChat, Open WebUI |
| 모델 실행 | Ollama, vLLM, TGI |
| 오픈소스 모델 | Llama, Qwen, Mistral, DeepSeek 계열 일부 |
| 도구 표준화 | MCP 서버 |
| 메시지 버스 | Kafka |
| 수집/라우팅 | NiFi |
| 실시간 처리 | Flink |
| 배치 처리 | Spark |
| 오케스트레이션 | Airflow |
| 운영 DB | PostgreSQL, MySQL |
| 상태 캐시 | Redis |

## 6. 범용 AI 데이터파이프라인의 핵심 입력 구조

자연어를 직접 코드로 연결하지 말고 중간 spec으로 변환해야 한다.

### 6-1. 예시 자연어 요청

> "이 폴더의 CSV를 읽어 표준 거래 스키마로 변환하고 Kafka에 넣은 뒤, 컨슈머가 읽어서 MySQL 테이블에 적재해라."

### 6-2. 구조화된 spec 예시

```yaml
pipeline:
  name: csv_to_kafka_mysql

source:
  type: csv_folder
  path: data/incoming
  format: csv

transform:
  schema: standard_transaction_v1
  mode: strict
  primary_key: transaction_id
  reject_topic: platform.events.dlq

transport:
  type: kafka
  topic: platform.events.standardized
  partitions: 6
  replication_factor: 3

sink:
  type: mysql
  table: transactions
  write_mode: upsert

validation:
  required_fields:
    - transaction_id
    - user_id
    - amount
    - event_time
```

이렇게 해야 특정 프로젝트 이름이나 도메인에 종속되지 않고, 고객별로 재사용 가능한 플랫폼이 된다.

## 7. 핵심 구성 요소별 역할

### 7-1. Kafka

Kafka는 범용 AI 데이터파이프라인의 이벤트 버스 역할을 맡는다.

- 수집 단계와 처리 단계 분리
- 표준 스키마 메시지 공유
- DLQ 토픽 분리
- 소비자 그룹별 독립 처리
- 이벤트 기반 파이프라인 결합

범용 플랫폼에서 Kafka는 거의 필수적인 중심 계층에 가깝다.

### 7-2. NiFi

NiFi는 다중 소스 수집과 라우팅의 핵심 엔진이다.

- CSV 파일 수집
- REST API polling
- DB polling 및 증분 추출
- schema transform
- Kafka publish
- 품질 실패 라우팅

범용 플랫폼에서 NiFi는 `수집 adapter engine` 역할을 수행한다.

### 7-3. Flink

Flink는 실시간 처리 엔진이다.

- 실시간 집계
- 상태 기반 처리
- 실시간 품질 검증
- 이상 탐지
- Kafka -> Kafka/DB 실시간 경로

범용 플랫폼에서는 "실시간 의사결정 계층"으로 사용할 수 있다.

### 7-4. Spark

Spark는 배치 처리와 대용량 정제/재처리에 적합하다.

- 배치 ETL
- backfill
- JDBC 적재
- 대용량 정합성 검증
- Delta Lake 적재

범용 플랫폼에서는 "배치 변환 및 대용량 적재 계층"이다.

### 7-5. Airflow

Airflow는 핵심 데이터 오케스트레이션 엔진이다.

- DAG 기반 종속성 제어
- 배치 파이프라인 실행
- 재시도, 재실행, 스케줄링
- 검증 작업 orchestration
- Spark, Python, shell, DB 작업 연결

범용 AI 데이터파이프라인에서는 Airflow가 핵심 운영 orchestration 계층에 해당한다.

### 7-6. DB

DB는 source와 sink 모두에 쓰인다.

- PostgreSQL: 메타데이터, 운영 상태, 일부 소스/타겟
- MySQL: 레거시 소스 또는 적재 타겟
- 기타 RDBMS: 고객 환경 확장 가능

범용 플랫폼에서는 DB를 adapter 구조로 다루는 것이 맞다.

### 7-7. Redis

Redis는 파이프라인 보조 상태 계층으로 적합하다.

- 진행 상태 캐시
- deduplication key
- feature flag
- 빠른 상태 조회
- 임시 작업 상태 공유

Redis는 핵심 처리 엔진이라기보다 상태 보조 계층이다.

## 8. MCP 서버 활용 방안

### 8-1. MCP 서버의 역할

MCP 서버는 각 시스템의 API, CLI, 운영 모델 차이를 감추고 AI가 공통 도구로 사용할 수 있게 한다.

즉 AI는 다음과 같은 질문을 표준 도구 호출로 바꿀 수 있다.

- "Kafka 토픽의 ISR 상태를 보여줘"
- "Airflow에서 지난 24시간 실패한 DAG를 보여줘"
- "NiFi queue backlog가 큰 프로세서 그룹을 찾아줘"
- "Flink job 중 checkpoint 실패가 있는가"
- "Spark에서 실패한 job이 있는가"

### 8-2. 시스템별 MCP 서버 역할

| MCP 서버 | 최소 기능 |
|------|------|
| Kafka MCP | 토픽 목록, 토픽 상세, 리더/ISR, consumer lag, cluster health |
| NiFi MCP | process group 상태, processor 상태, queue backlog, bulletin, provenance 요약 |
| Airflow MCP | DAG 목록, DAG run, 실패 task, 상태 요약 |
| Flink MCP | job 목록, job 상태, checkpoint, 예외 요약 |
| Spark MCP | app 목록, stage/job/executor 상태, 실패 작업 조회 |
| DB MCP | 테이블 존재 여부, row count, schema 조회, 샘플 데이터 조회 |
| Redis MCP | key 상태, TTL, 캐시된 파이프라인 상태 조회 |

### 8-3. 구현 우선순위

범용 플랫폼 관점의 현실적 우선순위는 다음과 같다.

1. Kafka MCP
2. Airflow MCP
3. NiFi MCP
4. DB MCP
5. Redis MCP
6. Flink MCP
7. Spark MCP

즉 `운영 가시성이 높은 시스템부터` 붙이는 것이 맞다.

## 9. LibreChat / Open WebUI 활용 방안

대화형 UI는 핵심 처리 엔진이 아니라 사람과 AI를 연결하는 운영 인터페이스다.

### 역할

- 자연어 목표 입력
- MCP 서버 호출
- 파이프라인 상태 질의
- 장애 원인 설명
- 승인 전 영향도 설명
- 실행 계획 요약

### 적합한 이유

- 오픈소스 기반
- 멀티 모델 연동 가능
- MCP 및 tool 호출 계층과 자연스럽게 결합
- 향후 Admin Portal 전 대화형 진입점 역할 가능

### 권장 위치

- OpenClaw: 상위 실행/에이전트 런타임
- LibreChat / Open WebUI: 대화형 운영 인터페이스

## 10. 실제 구현 전략

### 10-1. 권장 전략

처음부터 완전 자율형 플랫폼을 만들기보다 아래 단계로 접근해야 한다.

#### 1단계: Template 기반 PoC

- CSV -> Kafka template
- Kafka -> DB sink template
- 표준 schema template
- validation template

#### 2단계: Spec 기반 생성

- 자연어 -> spec
- spec -> 코드 / 설정 / DDL / 검증 스크립트 생성

#### 3단계: MCP 기반 운영 가시성

- Kafka, Airflow, NiFi부터 MCP 서버 구현
- 상태를 AI가 설명 가능하게 함

#### 4단계: OpenClaw + 대화형 UI 연결

- OpenClaw가 spec 생성 및 실행 계획 제안
- LibreChat/Open WebUI가 운영 질의응답 제공

#### 5단계: 승인형 운영 변경

- diff 확인
- validation 통과
- 승인 후 적용

### 10-2. 가장 현실적인 PoC 범위

범용 플랫폼 PoC는 아래 수준으로 시작하는 것이 적절하다.

- 입력: 단일 CSV 폴더 또는 단일 REST API
- 표준 schema: 1종
- Kafka 토픽: 1개
- sink: MySQL 또는 PostgreSQL 1개 테이블
- producer/consumer: Python 기반
- 운영 가시성: Kafka MCP
- 상위 인터페이스: OpenClaw 또는 LibreChat 중 1개

### 10-3. 예상 기간

| 범위 | 예상 기간 |
|------|----------|
| 좁은 동작형 PoC | 3~5일 |
| 데모 가능한 안정형 PoC | 1~2주 |
| spec + MCP + 대화형 운영 포함 PoC | 2~4주 |
| 운영형 MVP | 4~8주 이상 |

## 11. 주요 리스크와 대응

### 11-1. 주요 리스크

- 입력 포맷이 매번 달라지면 schema mapping 자동화가 어려움
- 자연어 요구가 모호하면 잘못된 spec이 생성될 수 있음
- sink DB upsert 정책이 불명확하면 중복/유실 위험
- AI가 생성한 구성이 실행은 되지만 운영에는 취약할 수 있음
- 승인 없는 제어 기능은 위험함

### 11-2. 대응 전략

- 자유형 생성보다 template + spec 기반 생성
- spec 검토 단계 도입
- validation script 자동 생성
- DLQ와 reject 구조 표준화
- 제어 기능은 승인형으로 제한

## 13. 이 아키텍처로 할 수 있는 10가지 예시

1. 특정 폴더의 CSV를 읽어 표준 거래 schema로 변환하고 Kafka 토픽에 적재한 뒤 DB에 자동 적재
2. REST API에서 일정 주기로 이벤트를 수집해 Kafka로 전송하고, AI가 이벤트 구조와 오류 패턴을 설명
3. PostgreSQL 고객 마스터 변경분을 수집해 Kafka로 전달하고 Spark가 주기적으로 정제 테이블 갱신
4. Kafka consumer lag가 급증했을 때 MCP가 상태를 수집하고 OpenClaw가 원인 후보를 제시
5. Airflow DAG가 실패하면 MCP를 통해 실패 task와 상태를 요약하고 운영자에게 복구 절차를 제안
6. NiFi queue backlog가 증가하면 병목 프로세서를 식별하고 변경 전 영향도를 설명
7. Flink checkpoint 실패를 감지해 운영자가 즉시 영향 범위와 재시작 판단을 할 수 있게 지원
8. Redis에 저장된 상태를 기반으로 현재 파이프라인 진행률과 재시도 상태를 설명
9. 자연어로 "이 CSV를 이 포맷으로 적재해"라고 지시하면 AI가 spec을 만들고 template 기반 producer/consumer/DDL/검증 스크립트를 생성
10. 여러 고객 환경에서 source/sink adapter만 바꿔 같은 플랫폼 구조를 재사용

## 14. 참조 시나리오의 위치

현재 저장소의 Nexus Pay 시나리오는 이 플랫폼의 참조 검증 모델로 유용하다.

활용 방식:

- Kafka 장애/복구 시나리오 검증
- NiFi 수집 흐름 검증
- Airflow 오케스트레이션 검증
- Spark/Flink 처리 경로 검증

하지만 플랫폼 문서와 구현은 Nexus Pay 전용이 아니라 고객·도메인 독립적인 구조를 유지해야 한다.

## 15. 최종 권고

범용 오픈소스 AI 데이터파이프라인 플랫폼을 구축하려면 다음 원칙을 지켜야 한다.

- 특정 도메인보다 spec과 adapter 중심으로 설계할 것
- 상용 코딩 에이전트보다 오픈소스 런타임과 MCP 중심 구조로 갈 것
- OpenClaw는 상위 실행 콘솔, LibreChat/Open WebUI는 대화형 운영 인터페이스로 사용할 것
- 핵심 데이터 처리 엔진은 Kafka, NiFi, Flink, Spark, Airflow가 맡도록 할 것
- 운영 가시성은 MCP 서버 계층에서 먼저 확보할 것
- 승인과 검증 없는 자동 변경은 허용하지 말 것

가장 현실적인 시작점은 다음과 같다.

- CSV/API -> Kafka -> DB 적재의 좁은 PoC
- Kafka MCP 우선 구현
- Airflow/NiFi MCP 확장
- OpenClaw와 LibreChat/Open WebUI 연결

이 접근이 범용성과 구현 가능성을 동시에 만족시키는 가장 좋은 방향이다.
